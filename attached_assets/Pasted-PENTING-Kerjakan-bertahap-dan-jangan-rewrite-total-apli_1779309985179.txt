PENTING: Kerjakan bertahap dan jangan rewrite total aplikasi.

Sebelum membuat perubahan:
1. Analisa struktur project existing.
2. Cek route existing.
3. Cek database/schema existing.
4. Cek auth/login existing.
5. Cek module RFQ/order/vendor/customer yang sudah ada.
6. Gunakan kode existing jika masih relevan.
7. Jangan duplicate fitur.
8. Jangan membuat dua flow aktif yang saling bentrok.

Tujuan utama:
Refactor flow RFQ lama agar tidak lagi customer langsung blast ke vendor.

Flow baru wajib:
Customer → Admin → Vendor → Admin → Customer → Order Execution

Jika ditemukan logic lama:
Customer → Vendor langsung

Maka:
- nonaktifkan logic tersebut
- hapus duplicate route/API/component yang membuat konflik
- migrasikan ke flow baru
- gunakan flow baru sebagai single source of truth

Jangan menghapus module lain yang tidak terkait, seperti:
- POS
- inventory
- warehouse
- accounting
- product
- auth
- company management

==================================================
TAHAP 1 — REFACTOR RFQ FLOW
==================================================

Saat customer membuat request:
- jangan kirim langsung ke vendor
- simpan sebagai RFQ baru
- status = admin_review
- tampilkan di dashboard admin RFQ

Admin RFQ Detail:
Buat/perbaiki halaman admin RFQ detail agar admin bisa:
- melihat detail request customer
- memilih beberapa vendor
- menentukan response deadline
- klik Blast ke Vendor

Saat admin klik Blast ke Vendor:
- buat token unik untuk setiap vendor
- buat link vendor:
  /vendor-form/:token
- status RFQ = vendor_blasted
- status vendor = waiting_response

==================================================
TAHAP 2 — VENDOR MINI FORM
==================================================

Buat mini form ringan dan mobile-friendly:
 /vendor-form/:token

Vendor form harus:
- token based access
- tidak wajib login dashboard
- cepat dibuka dari WhatsApp
- hanya fokus untuk isi penawaran

Tampilkan:
- RFQ number
- service type
- origin
- destination
- cargo detail
- harga basic / target price
- response deadline
- countdown timer

Vendor dapat memilih:
1. Terima Harga Basic
2. Ajukan Harga Baru
3. Tolak

Jika Terima Harga Basic:
- vendor_response_price = basic_price
- status = accepted_basic_price

Jika Ajukan Harga Baru:
- wajib isi harga baru
- wajib isi ETA
- wajib isi catatan
- status = counter_offer

Jika Tolak:
- status = rejected
- alasan optional

Semua jawaban vendor harus masuk ke RFQ yang sama.

TAHAP 3 — ADMIN VENDOR COMPARISON MINI FORM
==================================================

Buat 1 halaman comparison untuk setiap RFQ:
 /admin-vendor-comparison/:rfq_id

PENTING:
- 1 RFQ hanya boleh punya 1 comparison page
- jangan buat form comparison baru setiap vendor submit
- vendor response baru harus mengupdate halaman yang sama

Tampilkan:
- RFQ number
- customer
- service type
- origin
- destination
- deadline
- total vendor dikirim
- sudah jawab
- belum jawab
- rejected
- expired
- counter offer

Tampilkan daftar vendor:
- nama vendor
- harga basic
- harga penawaran
- ETA
- status
- catatan
- attachment jika ada
- waktu submit/update
- label NEW jika baru update
- tombol Pilih Vendor Ini
- tombol Minta Revisi
- tombol Tolak Semua

Status vendor:
- waiting_response
- accepted_basic_price
- counter_offer
- rejected
- expired
- selected
- not_selected
- late_response

Jika vendor belum menjawab sampai deadline:
- ubah status menjadi expired

Jika vendor submit setelah admin sudah memilih vendor:
- simpan sebagai late_response
- jangan ubah vendor terpilih otomatis

Tambahkan auto-refresh setiap 5-10 detik atau realtime subscription jika tersedia.

TAHAP 4 — WHATSAPP NOTIFICATION RECAP LIST
==================================================

Saat vendor submit atau update penawaran:
1. simpan response ke database
2. ambil semua vendor response untuk RFQ yang sama
3. ambil vendor yang belum menjawab
4. kirim WhatsApp ke admin dalam format recap list
5. link comparison harus tetap sama

Format WA admin:

🔔 Update Penawaran Vendor

RFQ: {rfq_number}
Layanan: {service_type}
Rute: {origin} → {destination}

Daftar penawaran:
1. {vendor_name} — Rp{price} — ETA {eta} — {status}
2. {vendor_name} — Rp{price} — ETA {eta} — {status}
3. {vendor_name} — Rp{price} — ETA {eta} — {status}

Belum jawab:
- {vendor_name}
- {vendor_name}

Lihat & pilih vendor:
{admin_comparison_link}

Ketentuan:
- WA boleh terkirim setiap ada update
- tetapi isinya harus recap list terbaru
- jangan hanya vendor terakhir
- jangan buat link baru
- semua WA mengarah ke /admin-vendor-comparison/:rfq_id
- jangan membuat comparison form baru

- margin
- comparison vendor

Customer quote mini form harus:

Customer hanya melihat:
- RFQ number
- service type
- origin

- catatan admin

Customer tidak boleh melihat:
- nama vendor
- harga vendor
- margin
- data internal admin

Customer dapat memilih:
1. Setuju
2. Minta Revisi
3. Tolak

Jika Setuju:
- status RFQ = customer_approved
- otomatis buat order
- status order = order_confirmed
- kirim notifikasi ke admin

- wajib isi catatan revisi

- jangan buat order

Jika Tolak:
- alasan optional
- status RFQ = customer_rejected
- kirim notifikasi ke admin
- jangan buat order

==================================================
SECURITY
==================================================

- Semua mini form memakai token valid
- Token expired tidak bisa submit
- Customer token hanya bisa lihat data customer
- Vendor token hanya bisa lihat RFQ vendor tersebut
- Admin comparison hanya untuk admin atau token admin valid
- Jangan expose vendor cost ke customer
- Jangan expose margin ke customer
- Jangan expose comparison vendor ke customer
- Simpan audit log untuk semua aksi penting

==================================================
ANTI CONFLICT RULES
==================================================

Jika ada conflict dengan flow lama:
- hapus logic lama yang customer langsung blast ke vendor
- hapus duplicate route/API/component yang menyebabkan double flow
- gunakan flow baru sebagai single source of truth
- jangan membuat dua RFQ system aktif bersamaan
- jangan membuat duplicate comparison page
- jangan membuat duplicate vendor response logic
- jangan membuat duplicate customer quotation logic

Jika ada file lama yang tidak digunakan dan menyebabkan conflict:
- cleanup / remove deprecated code
- pastikan build tetap sukses
- pastikan route utama tidak rusak

Jangan rewrite seluruh aplikasi.
Kerjakan modular, aman, dan bertahap.
